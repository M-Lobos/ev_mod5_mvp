
*** MODELO MVP ***
En MVP, la Vista y el Presenter no se conocen directamente; se comunican a través de una Interface.
    - Qué hace cada elemento
    Vista (ej: showProducts(list), MapsToDetail(product))
    Presenter (ej: onProductClicked(product)).
    Modelo (ej: val id:int, val  nombre : String, etc )

    La actividad, requiere una lista dinámica en la vista

Pasos lógicos
    1) Configurar el RV en el homePresenter (que será invocado en el MainActivity)
    2) Crear un ADAPTER que infle el diseño
         Implementar un clickListener para que el FRAGMENT sepa qué zapato se clicó

    3) Conexión de pantallas
        configurar el paso de datos, debe pasar el producto completo

    4) Persistencia en el carrito


NOTA 1; De los modelos:
En el concepto, shoes shoes es el modelo de datos del producto(data class), mientras que Cartmanager
es el modelo de lógica para los datos (repositorio). El presenter implementa la lógica que comunica
Soes con la view. Así En el MVP, la interfaz es quien posee el contrato del qué hacer (mas no como)

En el mvp la interfaz en sí es el contrato

NOTA 2; Del diseño:
El shoe_card_layout.xml es el diseño de la tarjeta para cada producto que será desplegado en el HomeFragment
El fragment_home.xml, es la "repisa"donde se despliegan las tarjetas para ser exhibicas en el homefragment
    -> El RV es, visualmente el fragment, en la lógica sigue siendo su contenedor

        * LayOut del RV (fragment_home.xml):
        -> android:id="@+id/rvShoes"
            Este es el nombre del RV para efectos del código

        -> app:layoutManager="androidx.recyclerview.widget.GridLayoutManager"
            Le dice a Android que se desea una cuadricula (grid) en lugar de una lista en el RV.

        -> app:spanCount="2"
        Define 2 "columnas" para el despliegue de las tarjetas.

        -> tools:listitem="@layout/shoe_card_layout":
        Esta línea es vital, es la que invoca y despliega las tarjetas en el RV


    *** Jerarquía de diseño ***
    -> MainActivity: edificio.
    -> HomeFragment: piso de ventas.
    -> RecyclerView: Las estanterías que llenan el piso.
    -> shoe_card_layout:  cada caja de zapatos.

NOTA 4: DE LOS ERRORES
Cuando el IDE dice "Overrides nothing" o "Class is not abstract and does not implement abstract member",
es porque el Contrato dice una cosa y la clase está haciendo otra diferente (falta o se cambió un parámetro
o falta una función).
